## 1. プロジェクト概要

### 1.1 プロダクト名

**ココロバランス(KokoroBalance)**

### 1.2 キャッチコピー

> 心の支えは、1本より3本。
> あなたの心は、何で満たされていますか?

### 1.3 プロダクトコンセプト

「自分の心が何によって支えられているか」を可視化し、支えの柱を増やし・育てることで、長期的に心を安定させる**セルフマネジメントツール**。

投資の分散理論をメンタルヘルスに応用した「心のポートフォリオ」を中核概念とするが、ユーザー体験としては**「依存を警告する」のではなく「心の柱を育てる」**ポジティブな枠組みで提供する。

**本アプリは医療機器・医療サービスではなく、診断・治療を目的としない**(→ 7章 安全性要件)。

### 1.4 検証したい仮説(このプロダクトが成立する条件)

v2では、機能を作る前提となる仮説を明示し、リリース計画(11章)の各段階で検証する。

| # | 仮説 | 検証方法 | 検証フェーズ |
|---|---|---|---|
| H1 | 「自分の心の支えの偏りを知りたい」というニーズが存在する | LP+ウェイトリスト登録率、オンボーディング診断の完了率 | β前〜β |
| H2 | 10秒記録なら1週間続けられる | 7日継続率(週4日以上記録) | β |
| H3 | ポートフォリオの可視化に「なるほど」体験がある | 診断直後のアンケート/オンボーディング離脱率 | β |
| H4 | AIコーチに月額を払う価値を感じる人がいる | お試し利用→Pro転換率 | 正式リリース |

**H1〜H3が棄却された場合、フェーズ2以降には進まずピボットを検討する。**

## 2. 解決する課題

### 2.1 課題の背景

人は恋人・推し・仕事・SNSなど特定の対象に心の支えが集中しやすい。その対象が不安定になった時、メンタルも大きく崩れる。複数の心の支えを持つ人ほど精神的に安定しやすい一方、「自分が何に支えられているか」は本人には気づきにくい。

### 2.2 競合分析

v1の「既存の気分記録アプリ」という抽象比較を廃し、実在サービスと比較する。

| サービス | 概要 | ココロバランスとの違い |
|---|---|---|
| Awarefy | AIメンタルパートナー。CBTベース、AIチャット、サブスク | 感情・思考の記録が中心。「何に支えられているか」の資産分布は扱わない |
| muute | AIジャーナリング。書いた内容をAIが分析 | 自由記述ベースで入力負荷が高い。分散という概念がない |
| Daylio | 絵文字ベースの気分トラッカー。超軽量入力 | 入力の軽さは目標とすべき水準。ただし気分の結果を記録するのみで、支えの構造は可視化しない |
| emol | AIと会話するメンタルアプリ | 会話中心。可視化・構造化は弱い |

**差別化の核:** 「感情の結果」ではなく「心を支える資産の分布と推移」を可視化する点。および事後対処ではなく予防(柱を増やす)にフォーカスする点。

**競合から学ぶ点:** Daylio級の入力の軽さがなければ土俵に立てない。AwarefyはAI+サブスクの価格受容性の先行事例(月額980〜1,600円帯)として参考にする。

### 2.3 リスク認識

- 予防価値は短期で知覚されないため、**短期的に毎日開く理由**(記録の手軽さ、記録直後の即時フィードバック、週次の発見)を別途設計する(4章)。
- 「依存」の指摘はユーザーを不快にさせうるため、警告ではなく育成のフレーミングで伝える(5.5節)。

## 3. ターゲットユーザー

### 3.1 一次ターゲット(MVP検証で狙う具体像)

v1の「20〜40代社会人」では広すぎるため、以下に絞ってLP訴求・βリクルーティングを行う。

> **20代後半〜30代の一人暮らし社会人で、生活が「仕事+特定の1つ(恋人または推し)」に集中している自覚がうっすらある人。**
> 過去に、その1つが揺らいだとき(失恋・推しの卒業・繁忙期)にメンタルが大きく崩れた経験がある。

- 利用シーン: 夜、寝る前にスマホで1日を軽く振り返る(10秒)
- 刺さる訴求: 「それがなくなったとき、あなたの心を支えるものは残っていますか?」

### 3.2 二次ターゲット(将来拡張)

- SNS疲れを感じている人
- 仕事一辺倒でワークライフバランスを見直したい人
- メンタルの浮き沈みを予防的にケアしたい人

※ 現在治療中・重度の抑うつ状態の人は本アプリの対象外とし、適切な相談先を案内する(7章)。

## 4. コアUX

### 4.1 体験フロー

```
【初日】オンボーディングで「この1ヶ月の振り返り診断」(3分)
        ↓
    その場で「現在の心のポートフォリオ」が完成 ← Ahaモーメント①
        ↓
【毎日】寝る前に10秒記録(タップ+3段階)
        ↓
    記録直後に今日のひとことフィードバック ← 毎日開く理由
        ↓
【毎週】週間レポートで変化と発見 ← Ahaモーメント②
        ↓
    「次に育てる心の柱」の提案 → 小さな行動 → 柱が増える
```

### 4.2 Ahaモーメントの定義(要件)

| モーメント | 体験 | 要件 |
|---|---|---|
| ① 診断直後 | 「自分は◯◯に◯%支えられているのか」という発見 | オンボーディング完了時点で必ず意味のあるポートフォリオが表示されること(コールドスタート禁止) |
| ② 週次 | 「先週と比べて柱が育っている/揺れた原因が見える」 | 週間レポートは記録が2日以上あれば生成されること |

### 4.3 記録体験の性能要件

- 日次記録は**アプリを開いてから保存まで10〜20秒**で完了できること(タップ数: カテゴリ選択+段階選択+保存で最大10タップ程度)
- 記録直後に即時フィードバック(1行)を必ず返すこと

## 5. 機能要件

### 5.1 MVP(フェーズ1)— 5機能に絞る

> **v1から削除・統合した機能:**
> - 「心の残高(デイリースコア)」→ 記録直後フィードバックとダッシュボード表示に簡素化(独立機能としては廃止)
> - 「心の資産スコア(資産スター)」→ 「心の柱」表示に統合(5.4)
> - 「分散指数スコア(HHI)」→ ユーザー向けには「柱の本数」で表現。HHIは内部指標としてのみ保持
> - 0〜100スコア入力 → 3段階に変更

#### 5.1.1 ユーザー登録・オンボーディング診断

- メールアドレス/Googleアカウントによるサインアップ・サインイン(Supabase Auth)
- ニックネーム設定
- **オンボーディング(3ステップ、合計3分以内):**
  1. コンセプト説明(1画面、スキップ可)
  2. 心のカテゴリ選択(プリセットから3〜8個。カスタム追加可)
  3. **振り返り診断:** 「この1ヶ月、それぞれがどれくらいあなたの心を満たしましたか?」を選んだカテゴリごとに3段階で回答
- 診断完了と同時に**初期ポートフォリオを表示**する(Ahaモーメント①)。初期診断データは `is_baseline` フラグ付きで保存し、日次記録が14日分蓄積されるまで集計にブレンドする(徐々に実データへ移行)

#### 5.1.2 心のカテゴリ設定

v1と同一のプリセット体系(人/趣味/推し/自己成長/健康/仕事/お金 + カスタム)。

- 有効カテゴリは**3〜8個を推奨**(多すぎると記録負荷が上がるため、9個以上はUI上で注意を表示)
- カテゴリの追加・無効化・並び替え・色変更

#### 5.1.3 毎日の記録(10秒記録)

- 今日心が満たされたカテゴリを**タップで複数選択**
- 各カテゴリの満たされ度は**3段階**(すこし +1 / 満たされた +2 / とても +3)。デフォルトは +2 で、タップ数を最小化
- **0〜100スコアは廃止**(毎日再現性のある判断ができず入力負荷になるため)
- **心が揺れた出来事(任意):** ネガティブイベントは満足度と別枠で記録する
  - カテゴリ + 揺れの大きさ(小/中/大)+ 任意メモ
  - **割合(ポートフォリオ)計算には算入しない**。時系列上の「揺らぎイベント」として表示し、週間レポートの分析対象とする(→ C-2 数学的欠陥の解消)
- 記録は1日1回をベースに、同日再編集は上書き(upsert)。`daily_records` の (user_id, recorded_date) ユニーク制約と整合
- 保存直後に**今日のひとことフィードバック**を表示(例: 「今日は3本の柱があなたを支えました 🌱 友達の柱が今週2回目です」)。テンプレートベースで実装(AI不使用、無料機能)

#### 5.1.4 心のポートフォリオ + 心の柱(メイン画面)

**2つの指標を分離して表示する(v1の「割合のみ」の欠陥を解消):**

| 指標 | 意味 | 表示 |
|---|---|---|
| **バランス(相対)** | 心の支えの分布 | 円グラフ(カテゴリ別シェア%) |
| **充足度(絶対)** | どれだけ満たされているか | 期間合計ポイントと推移(バーグラフ/折れ線) |

- 集計期間: 7日 / 30日 / 90日 切り替え
- **計算式(要件として明記):**
  - カテゴリ別シェア% = カテゴリ別ポジティブポイント合計 ÷ 全カテゴリのポジティブポイント合計 × 100
  - 揺らぎイベント(ネガティブ)はシェア計算に含めず、グラフ上にマーカー表示
  - 内部指標としてHHI(Σシェア²)を算出・保存(アラート判定・分析用。ユーザーには直接見せない)
- **心の柱表示:** 直近30日で「週1回以上記録されているカテゴリ」を「柱」として本数表示(例: 「いまのあなたの柱: 3本 🏛️🏛️🏛️」)。v1の資産スターの後継。柱の成長段階(芽→若木→柱)で継続を可視化し、育成の動機づけとする

#### 5.1.5 育成提案(旧: 偏りアラート)

**フレーミングを「警告」から「育成」に転換する。**

- **発動条件(データ下限を新設):** 記録日数が7日以上 かつ 記録件数が10件以上の場合のみ判定(初日の誤警告を防止)
- 判定: 単一カテゴリのシェアが60%以上、または柱が1本以下
- **メッセージはポジティブフレーミング:**
  > 🌱 いま、あなたの心は「恋人」が大きく支えてくれています。それ自体は素敵なことです。
  > もう1本、小さな柱を育てておくと、心はもっと安定します。
  > 最近ごぶさたの「読書」、今週5分だけ触れてみませんか?
- 提案するカテゴリは「過去に記録があるが最近減っているもの」を優先(ゼロからの新習慣より再開のほうが行動障壁が低い)
- アラートの非表示(ミュート)設定を可能にする(不快に感じるユーザーへの逃げ道)

#### 5.1.6 週間レポート

- 毎週月曜日に自動生成(記録が週2日以上ある場合)。任意タイミングでの再閲覧可
- 内容:
  - カテゴリ別シェアと前週比
  - 充足度合計と前週比
  - 揺らぎイベントの振り返り(「今週、心が揺れたのは◯◯でした」)
  - 来週の小さな提案1つ(無料: テンプレートベース / Pro: AI生成のパーソナライズコメント)
- v1レポート例にあった「睡眠不足が記録されています」のような、**入力仕様から導出できない内容は記載しない**(記録データから導出可能な事実のみ)

#### 5.1.7 AIコーチ(チャット形式)

- ユーザーが近況をテキスト入力し、AIがポートフォリオデータを文脈として具体的アドバイスを返す
- **プラン制御(v1から変更 — 決定事項2「Pro限定」の枠内で無料お試しを追加):**
  - 無料: **月3回まで**利用可(価値を体験させて転換につなげる)
  - Pro: 無制限 + 週間レポートAIコメント + 会話履歴の長期保持
- AI応答の品質要件: 「ポートフォリオを見れば分かること」の復唱ではなく、①共感 → ②データに基づく気づき → ③今週できる具体的な行動1つ、の構成とする
- **安全ガードレール必須**(7章): 診断・医療的助言の禁止、クライシス検知時の相談窓口案内
- 実装: Vertex AI (Claude)。システムプロンプトにポートフォリオ集計を付与(basic-design 7章を踏襲)

## 6. 通知・リマインド戦略(新設)

日次記録アプリの生命線であるため、独立要件とする。

| 手段 | 対象 | MVP |
|---|---|---|
| メールリマインド | 全ユーザー(Supabase/Resend等で送信) | ✅ 必須 |
| Web Push(PWA) | Android Chrome等の対応環境。**iOS SafariはPWAホーム画面追加時のみ動作**する制約を明記し、iOSユーザーにはホーム画面追加を案内するUIを用意 | ✅ 対応環境のみ |
| LINE通知連携 | 検討(日本のターゲットには最も確実) | フェーズ2 |

- リマインド時刻はユーザー設定(デフォルト21:30)
- 記録が3日途切れたユーザーには文言を変えた復帰メール(「おかえりなさい」トーン。罪悪感を煽らない)
- 通知はワンクリックでオフにできること

## 7. 安全性・倫理要件(新設・リリース必須)

メンタルヘルス領域のプロダクトとして、以下を**リリースブロッカー(未実装ならリリース不可)**とする。

### 7.1 非医療の明示

- 本アプリは医療機器・医療サービスではなく、診断・治療・医学的助言を行わないことを、利用規約・オンボーディング・AIコーチ画面に明示する
- 医療機器プログラム該当性を公開前に確認する(セルフケア・可視化ツールの範囲に留め、疾病の診断・治療・予防を目的とする表現を使わない。LPやストア文言も同基準)

### 7.2 クライシス対応

- **検知:** AIコーチへの入力・記録メモに希死念慮・自傷を示唆する表現が含まれる場合を検知する(キーワード+LLM判定の併用)
- **応答:** AIは通常の助言を中断し、共感を示した上で専門相談窓口を案内する
  - 案内先(日本): よりそいホットライン、いのちの電話、こころの健康相談統一ダイヤル 等
- 設定・ヘルプ画面に相談窓口一覧を常設する
- AIコーチのシステムプロンプトに、診断禁止・薬剤助言禁止・クライシス時のエスカレーション手順を含める

### 7.3 データの取り扱い

- 心の状態に関するデータは機微情報として扱い、第三者提供・広告利用をしない
- **AIコーチ利用時にポートフォリオデータが外部AI(Vertex AI)へ送信されることをプライバシーポリシーに明示し、初回利用時に同意を取得する**
- データエクスポート・アカウント削除(完全消去)機能を提供(個人情報保護法・GDPR対応)
- プライバシーポリシー・利用規約はβ公開前に整備する(v1では「未着手」だったが、βの前提条件に格上げ)

### 7.4 対象年齢

- 利用は16歳以上とする(利用規約に明記)

## 8. 非機能要件

### 8.1 パフォーマンス

- ホーム画面の初期表示: 2秒以内
- 日次記録の保存応答: 1秒以内(10秒記録の体験を壊さない)
- グラフ描画: 1秒以内

### 8.2 セキュリティ

- 通信はHTTPS(TLS 1.2以上)
- 認証はSupabase Auth(パスワードハッシュはSupabase管理)。バックエンドはJWT検証+Supabase RLSの二重防御
- 機微データの扱いは7.3に従う

### 8.3 可用性・スケーラビリティ

- 稼働率目標: 99.5%
- 初期: 〜1,000ユーザー。10万ユーザーへのスケールを考慮した設計(現行構成: Vercel + Railway/Cloud Run + Supabase)

## 13. データモデル(v2差分)

basic-design.md のスキーマをベースに、以下を変更・追加する。

```
daily_record_items(変更)
├─ score: 3段階制に変更(1〜3を格納。旧: -100〜100)
└─ (ネガティブ値は廃止 → fluctuation_events へ分離)

fluctuation_events(新規)— 心が揺れた出来事
├─ id
├─ user_id
├─ category_id(NULL可: カテゴリ外の出来事)
├─ occurred_date
├─ magnitude('small' | 'medium' | 'large')
├─ note(任意テキスト)
└─ created_at

categories(追加)
└─ is_baseline_included: オンボーディング診断由来かのフラグ用途は
   baseline_assessments 側で管理

baseline_assessments(新規)— オンボーディング振り返り診断
├─ id
├─ user_id
├─ category_id
├─ level(1〜3)
└─ created_at

ai_usage(新規)— 無料枠管理
├─ user_id
├─ month(YYYY-MM)
└─ chat_count
```

- ポートフォリオ集計: 日次記録が14日未満の期間は baseline を重み付きでブレンド(重みは日次データ量に応じて線形減衰)
- HHI・柱本数はAPI応答で都度算出(weekly_reports にスナップショット保存)
