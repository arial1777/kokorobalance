# ココロバランス Web版 修正実装計画書

**バージョン:** 1.0
**作成日:** 2026-07-04
**ベース:** `docs/requirements-v2.md`(要件定義書 v2.0)
**対象:** 既存Webアプリ(Next.js + NestJS + Supabase)の v2 要件への改修
**スコープ外:** iOSネイティブアプリ(フェーズ2以降で別途計画)。ただしWebはモバイルファースト+PWAとしてスマホ体験を最優先する。

---

## 1. 目的

要件定義書 v1 ベースで実装済みの現行Webアプリを、v2 要件(10秒記録・コールドスタート解消・育成フレーミング・安全要件・通知・計測)に適合させる。本書は「何を・どのファイルで・どの順に」変更するかを定義する。

---

## 3. ワークパッケージ(WP)別 修正計画

### WP1: DBマイグレーション(全WPの前提)

**新規マイグレーション(`backend/db/migrations/` に追加、`all_migrations.sql` へ反映):**

```sql
-- 010: 3段階スコアへの移行
-- 既存データ変換(現状は開発データのみのため単純変換でよい):
--   score >= 67 → 3 / 34..66 → 2 / 1..33 → 1
--   score < 0   → fluctuation_events へ移送後、行を削除
--   ( -1..-33 → 'small' / -34..-66 → 'medium' / -67以下 → 'large' )
-- 変換後: CHECK (score IN (1, 2, 3)) に付け替え、daily_records.total_score を再計算

-- 011: fluctuation_events(心が揺れた出来事)
--   id / user_id / category_id(NULL可) / occurred_date / magnitude('small'|'medium'|'large') / note / created_at

-- 012: baseline_assessments(オンボーディング振り返り診断)
--   id / user_id / category_id / level(1..3) / created_at
--   UNIQUE (user_id, category_id)

-- 013: ai_usage(AIコーチ無料枠管理)
--   user_id / month('YYYY-MM') / chat_count、PK (user_id, month)

-- 014: profiles 拡張
--   suggestion_muted BOOLEAN DEFAULT false(育成提案ミュート)
--   ai_consent_at TIMESTAMPTZ(AI送信同意日時)
--   email_reminder_enabled BOOLEAN DEFAULT true

-- 015: event_logs(KPI計測)
--   id / user_id / event_name / properties JSONB / created_at

-- 016: weekly_reports 拡張
--   fulfillment_total INTEGER(充足度合計)/ pillar_count INTEGER / fluctuation_summary JSONB
```

**対応するTypeORMエンティティの追加・修正:** `records/daily-record-item.entity.ts`(score型コメント)、新規 `records/fluctuation-event.entity.ts`、`onboarding/baseline-assessment.entity.ts`、`coach/ai-usage.entity.ts`、`profile/profile.entity.ts`、`analytics/event-log.entity.ts`、`reports/weekly-report.entity.ts`。

**受け入れ条件:** マイグレーション適用後、既存の記録データが3段階に変換され、ネガティブ記録が fluctuation_events に移送されていること。本番未リリースのため後方互換は不要(開発DBは再構築でも可)。

### WP2: 10秒記録への刷新(§5.1.3, §4.3)

**バックエンド:**
- `records/dto/create-record.dto.ts`: `score` のバリデーションを `@IsIn([1, 2, 3])` に変更
- `records/records.service.ts`: upsert は現行ロジック流用。`total_score` は新スコア合計
- 新規 `records/fluctuations` エンドポイント:
  - `POST /records/fluctuations` (occurred_date, category_id?, magnitude, note?)
  - `GET /records/fluctuations?from=&to=`
  - `DELETE /records/fluctuations/:id`
- レスポンスに**即時フィードバック用データ**を追加: `POST /records` の応答に `{ todaysPillars: number, feedback: { categoryName, weeklyCount } }` 等(テンプレート文言の組み立てはフロント側)

**フロントエンド(`frontend/src/app/(app)/record/page.tsx` 全面改修):**
- スライダー廃止。カテゴリカードをタップで選択(デフォルト「満たされた +2」)、選択済みカードの再タップで 3段階トグル(+1/+2/+3)またはチップ選択
- 目標: **選択→保存まで最大10タップ・10〜20秒**
- 画面下部に折りたたみ式「心が揺れた出来事はありましたか?(任意)」セクション: カテゴリ(任意)+ 小/中/大 + メモ
- 保存後、ダッシュボード直行ではなく**フィードバック画面(またはフルスクリーントースト)**を挟む: 「今日は3本の柱があなたを支えました 🌱」等のテンプレート文言 → 自動でダッシュボードへ
- 記録保存の体感1秒以内(§8.1): mutation中のオプティミスティックUI

**受け入れ条件:** 3カテゴリの記録がタップのみで10秒以内に完了する。揺らぎ記録はポートフォリオの円グラフに影響しない。

### WP3: オンボーディング振り返り診断(§5.1.1、コールドスタート解消)

**バックエンド:**
- 新規 `POST /onboarding/baseline`: `[{ categoryId, level(1..3) }]` を baseline_assessments へ保存
- `portfolio/portfolio.service.ts` に**ベースラインブレンド**を実装:
  - 日次記録の記録日数 `d`(0〜14)に対し、重み `w = max(0, (14 - d) / 14)`
  - カテゴリ別ポイント = 日次実績ポイント + baseline_level × ブレンド係数 × w
  - `d >= 14` でベースライン寄与ゼロ(完全に実データへ移行)
  - レスポンスに `isBlended: boolean` を含め、フロントで「はじめの診断を含む」注記を表示

**フロントエンド(`frontend/src/app/onboarding/page.tsx`):**
- STEPS を「ようこそ → カテゴリ → **振り返り診断** → あなたのポートフォリオ」に変更
- 診断ステップ: 「この1ヶ月、それぞれがどれくらい心を満たしましたか?」を選択カテゴリごとに3段階で回答(1画面、カテゴリごとに3ボタン)
- 最終ステップ: **その場で初期ポートフォリオ(円グラフ)を表示**(Ahaモーメント①)。「これがいまのあなたの心のバランスです」→ ダッシュボードへ
- 所要時間3分以内(§5.1.1)。診断は全カテゴリ回答必須だがカテゴリ数は3〜8個に誘導済み

**受け入れ条件:** 新規登録ユーザーがオンボーディング完了時点で意味のある円グラフを見られる。登録初日に育成提案(旧アラート)が発動しない(WP4のデータ下限による)。

### WP4: ポートフォリオ再設計 — 充足度・柱・育成提案(§5.1.4, §5.1.5)

**バックエンド(`portfolio/portfolio.service.ts`):**
- レスポンス拡張:
  ```
  {
    breakdown: [...],            // 既存(シェア%)
    fulfillment: {               // 新規: 充足度(絶対量)
      total: number,             // 期間合計ポイント
      weeklyTrend: [{ weekStart, total }]
    },
    pillars: {                   // 新規: 心の柱
      count: number,             // 直近30日で週1回以上記録のあるカテゴリ数
      items: [{ categoryName, color, stage: 'sprout'|'young'|'pillar', weeklyFrequency }]
    },
    suggestion: {                // 旧alertを改名・転換
      exists: boolean,
      message: string,           // 育成フレーミング文言
      suggestedCategory?: {...}  // 最近減っているカテゴリを優先
    },
    diversityScore,              // 内部指標として維持(UIの主役からは降格)
    isBlended, totalRecordDays
  }
  ```
- **育成提案の判定(`buildAlert` → `buildSuggestion` に改修):**
  - 発動条件: `記録日数 >= 7 かつ 記録件数 >= 10`(§5.1.5 データ下限)
  - 判定: 最大カテゴリのシェア60%以上 または 柱1本以下
  - 文言をポジティブに全面変更(v2 §5.1.5 の例文をベース)。「依存」という単語を使わない
  - 提案カテゴリ: 「過去に記録があるが直近2週間で減っているカテゴリ」を優先、なければ未記録の有効カテゴリ
  - `profiles.suggestion_muted = true` の場合は `exists: false`
- 柱のステージ: 週1回以上×継続週数で 芽(1週)→若木(2〜3週)→柱(4週以上)

**フロントエンド:**
- `dashboard/page.tsx` 再構成:
  - ヒーローカード「今日の心の残高 +100」→ **「今日のあなたの柱」**(今日記録したカテゴリと柱の本数。数値スコアの合計は主役にしない)
  - 「偏りアラート」(amber警告) → **「育成提案」カード**(🌱、warningアイコン・警告色を廃止し緑系に)
  - 「分散指数スコア」カード → **「心の柱」カード**(柱の本数+ステージ表示)。HHIの数値表示は撤去
- `portfolio/page.tsx`: バランス(円グラフ)/充足度(週次バーグラフ)のタブまたは縦並び2セクション。揺らぎイベントを時系列マーカーで表示
- 設定画面に「育成提案を表示しない」トグル追加

**受け入れ条件:** 記録6日目まで提案が発動しない。恋人80%でも警告色・「依存」文言が出ない。柱の本数と成長段階が表示される。

### WP5: 週間レポート拡張(§5.1.6)

**バックエンド(`reports/reports.service.ts`):**
- **生成条件:** 対象週の記録日数が2日以上の場合のみ生成(cron・手動とも)。満たない場合は「今週は記録が少なかったので、来週また会いましょう」状態を返す
- 内容拡張(weekly_reports の新カラムへ保存):
  - カテゴリ別シェアと**前週比**(前週レポートとの差分計算)
  - 充足度合計と前週比
  - 柱の本数
  - 揺らぎイベントの振り返り(該当週の fluctuation_events 集計)
- **AIコメント(Pro):** cron生成時に plan='pro' のユーザーのみ Vertex AI でコメント生成し `ai_comment` へ保存。無料はテンプレート文の提案1つ(「最近減っている◯◯を、今週5分だけ」)
- 入力データから導出できない文言(「睡眠不足」等)を生成しないよう、AIプロンプトに「与えられた集計データに含まれる事実のみに言及する」制約を追加

**フロントエンド(`report/page.tsx`):** 前週比の増減表示、揺らぎ振り返りセクション、Pro向けAIコメント(無料ユーザーにはロックUI+アップグレード導線)

**受け入れ条件:** 記録1日の週はレポートが生成されず案内が出る。無料/Proで表示が出し分けられる。

### WP6: AIコーチ — 無料枠・品質・ガードレール(§5.1.7, §7.2)

**バックエンド(`coach/`):**
- **無料枠(月3回):**
  - `pro-plan.guard.ts` を `CoachQuotaGuard` に置換: Proは無制限、無料は ai_usage の当月 `chat_count < 3` なら許可しインクリメント
  - `GET /coach/quota` 新設: `{ plan, remaining, limit }`
- **プロンプト改修(`coach.service.ts`):**
  - 応答構成の指示を v2 §5.1.7 に合わせ「①共感 → ②データに基づく気づき → ③今週できる具体的行動1つ」に変更
  - 「診断・医療的助言・薬剤への言及の禁止」「データにない事実に言及しない」を明記
  - システムプロンプトへ渡すデータに柱・充足度・揺らぎイベントを追加(WP4/WP5の成果を利用)
- **クライシス対応(§7.2)【リリースブロッカー】:**
  - 新規 `coach/crisis-detector.service.ts`: ユーザー入力に対しキーワード検知(希死念慮・自傷の語彙リスト)+ LLM判定(キーワード検知が曖昧な場合)の2段構え
  - 検知時: 通常のAI応答をスキップし、共感+相談窓口(よりそいホットライン、いのちの電話、こころの健康相談統一ダイヤル)を案内する固定応答を返す。`ai_coach_messages` にはフラグ付きで保存
  - システムプロンプト側にも同様のエスカレーション指示を入れる(二重防御)
  - 検知ロジックは記録メモ(`note`)にも将来適用できるよう独立サービスにする
- **AI送信同意(§7.3):** 初回チャット時に `profiles.ai_consent_at` が未設定なら 428 等で同意要求 → フロントで同意モーダル → `POST /profile/ai-consent`

**フロントエンド(`coach/page.tsx`):**
- 無料ユーザー: ロック画面を廃止し、チャットUI+「今月あと◯回」バッジ。0回になったらアップグレード導線
- 初回利用時の同意モーダル(「会話内容とポートフォリオデータがAI(Vertex AI)に送信されます」)
- 画面フッターに「AIコーチは医療行為ではありません。つらいときは相談窓口へ」+窓口リンク常設

**受け入れ条件:** 無料ユーザーが月3回まで会話でき、4回目でアップグレード案内になる。クライシス表現のテスト入力に対し、助言ではなく窓口案内が返る。同意なしにAIへデータが送信されない。

### WP9: 計測基盤(§10 KPIファネル)

- `backend/src/analytics/` 新設: `POST /events`(event_name + properties)→ event_logs へ保存
- フロントに `lib/analytics.ts`(送信ヘルパー)を追加し、以下を計測:

| イベント | 対応KPI |
|---|---|
| `signup_completed` | 登録 |
